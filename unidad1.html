Introducción al Desarrollo Web Orientado a Servicios

Conceptos fundamentales, Arquitectura SOA y Estándares Web.

1. Paradigma del Desarrollo Orientado a Servicios

El desarrollo moderno se basa en la creación de aplicaciones distribuidas que colaboran entre sí. Este paradigma permite que sistemas dispares (construidos en diferentes lenguajes) se comuniquen a través de la red.

Servicios en la Nube

Para implementar estos servicios, es fundamental distinguir los modelos de entrega en la nube:

IaaS
Infraestructura

Alquiler de hardware virtual (servidores, redes). Tú administras el sistema operativo.

PaaS
Plataforma

Entorno listo para desarrollar y desplegar aplicaciones sin gestionar servidores.

SaaS
Software

Aplicaciones finales listas para usar a través del navegador (ej. Gmail, Office 365).

Aplicaciones Web Híbridas (Mashups)

Un Mashup es una aplicación web que utiliza contenido de más de una fuente para crear un servicio nuevo.
Ejemplo: Una web que toma datos de delincuencia de una API gubernamental y los muestra sobre Google Maps.

2. Arquitectura Orientada a Servicios (SOA)

SOA es un estilo arquitectónico donde las funcionalidades de software se agrupan en "servicios" lógicos e independientes. Cada servicio tiene un propósito específico y se comunica mediante protocolos estándar.

Principios de Diseño

  • Reutilización: El servicio puede ser usado por múltiples aplicaciones.
  • Bajo Acoplamiento: Los servicios son independientes; cambiar uno no rompe a los demás.
  • Abstracción: El usuario del servicio no necesita saber la lógica interna, solo cómo usarlo.

3. Estándares y Protocolos (Glosario)

Para garantizar la interoperabilidad, SOA depende de estándares estrictos. A continuación se definen los más importantes:

Es el lenguaje estándar para el intercambio de datos. Permite estructurar la información de forma legible tanto para humanos como para máquinas. Es la base sobre la que se construyen SOAP y WSDL.

Es un protocolo basado en XML para intercambiar información. Define las reglas de "cómo" se envía el mensaje, incluyendo seguridad y manejo de errores. Es muy riguroso y seguro.

Es un documento XML que funciona como el "manual de instrucciones" del servicio. Describe qué funciones están disponibles, qué datos se deben enviar y qué respuesta se recibirá.

Es como un "directorio telefónico" para servicios web. Permite a las empresas publicar sus servicios (WSDL) en un registro para que otros desarrolladores puedan encontrarlos y conectarse.

A diferencia de SOAP, REST no es un protocolo, sino un estilo de arquitectura. Utiliza los métodos estándar de HTTP (GET, POST, PUT, DELETE) y suele ser más ligero y rápido, prefiriendo JSON sobre XML.
